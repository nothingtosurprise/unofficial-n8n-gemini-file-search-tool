## Common Unit Test Patterns

### Pattern 1: Basic Operation Test (Happy Path)

**Use Case:** Testing successful execution of a node operation

**Example:** Store Creation Operation

```typescript
it('should create store with display name', async () => {
  // Arrange
  mockGetNodeParameter
    .mockReturnValueOnce('My Test Store'); // displayName

  const expectedResponse = {
    name: 'fileSearchStores/test-store-id',
    displayName: 'My Test Store',
    createTime: '2025-11-24T00:00:00Z',
  };

  (apiClient.geminiApiRequest as jest.Mock).mockResolvedValue(expectedResponse);

  // Act
  const result = await createStore.call(mockExecuteFunctions as IExecuteFunctions, 0);

  // Assert
  expect(result).toEqual(expectedResponse);
  expect(apiClient.geminiApiRequest).toHaveBeenCalledWith(
    'POST',
    '/fileSearchStores',
    { displayName: 'My Test Store' },
  );
});
```

**Key Elements:**
- **Arrange:** Set up mocks and test data
- **Act:** Execute the function
- **Assert:** Verify output and mock calls
- **One assertion focus:** Tests typically verify one behavior

### Pattern 2: Validation Test

**Use Case:** Testing input validation before API calls

**Example:** Store Name Validation

```typescript
it('should validate store name before deletion', async () => {
  // Arrange
  mockGetNodeParameter
    .mockReturnValueOnce('invalid-store'); // Invalid format

  (validators.validateStoreName as jest.Mock).mockImplementation(() => {
    throw new NodeOperationError(
      mockNode,
      'Invalid store name format. Must be fileSearchStores/{id}',
    );
  });

  // Act & Assert
  await expect(
    deleteStore.call(mockExecuteFunctions as IExecuteFunctions, 0),
  ).rejects.toThrow('Invalid store name format');

  expect(validators.validateStoreName).toHaveBeenCalledWith('invalid-store');
});
```

**Key Elements:**
- **Pre-condition validation:** Tests check input validation
- **Mock validator functions:** Validators are mocked to test logic
- **Error assertions:** Verify correct error is thrown

### Pattern 3: Optional Parameter Test

**Use Case:** Testing behavior with optional/conditional parameters

**Example:** Import File with Optional Display Name

```typescript
it('should import file with display name', async () => {
  // Arrange
  mockGetNodeParameter
    .mockReturnValueOnce('fileSearchStores/test-store') // storeName
    .mockReturnValueOnce('files/abc-123') // fileName
    .mockReturnValueOnce('Imported Document') // displayName (optional)
    .mockReturnValueOnce(false) // waitForCompletion
    .mockReturnValueOnce({}) // customMetadata
    .mockReturnValueOnce({}); // chunkingOptions

  (apiClient.geminiApiRequest as jest.Mock).mockResolvedValue({ name: 'op' });

  // Act
  await importFile.call(mockExecuteFunctions as IExecuteFunctions, 0);

  // Assert
  expect(apiClient.geminiApiRequest).toHaveBeenCalledWith(
    'POST',
    '/fileSearchStores/test-store:importFile',
    expect.objectContaining({
      fileName: 'files/abc-123',
      displayName: 'Imported Document',
    }),
  );
});
```

**Key Elements:**
- **Sequential mock returns:** Each `mockReturnValueOnce` represents one parameter
- **Conditional inclusion:** Verify optional fields are included when provided
- **Partial object matching:** Use `expect.objectContaining()` for partial matches

### Pattern 5: Polling/Async Operation Test

**Use Case:** Testing long-running operations with polling

**Example:** Document Upload with Completion Polling

```typescript
it('should wait for completion when requested', async () => {
  // Arrange
  mockGetNodeParameter
    .mockReturnValueOnce('fileSearchStores/test-store') // storeName
    .mockReturnValueOnce('files/abc-123') // fileName
    .mockReturnValueOnce('') // displayName
    .mockReturnValueOnce(true) // waitForCompletion
    .mockReturnValueOnce({}) // customMetadata
    .mockReturnValueOnce({}); // chunkingOptions

  const mockOperation = { name: 'operations/import-op', done: false };
  const mockCompleted = { name: 'documents/imported-doc', state: 'STATE_ACTIVE' };

  (apiClient.geminiApiRequest as jest.Mock).mockResolvedValue(mockOperation);
  (apiClient.pollOperation as jest.Mock).mockResolvedValue(mockCompleted);

  // Act
  const result = await importFile.call(mockExecuteFunctions as IExecuteFunctions, 0);

  // Assert
  expect(apiClient.pollOperation).toHaveBeenCalledWith('operations/import-op');
  expect(result).toEqual(mockCompleted);
});
```

**Key Elements:**
- **Polling simulation:** Mock returns `done: false` initially, then `done: true`
- **Operation state verification:** Verify that polling function is called with correct operation name
- **Result chaining:** Assert that final result comes from polling function

### Pattern 6: Error Handling Test

**Use Case:** Testing error conditions and exception handling

**Example:** API Error Handling

```typescript
it('should throw error when operation fails', async () => {
  // Arrange
  const errorDetails = {
    fieldErrors: [
      {
        field: 'fileName',
        description: 'File not found in Files API',
      },
    ],
  };

  (mockContext.helpers!.request as jest.Mock).mockResolvedValue({
    done: true,
    error: {
      code: 404,
      message: 'Import source not found',
      details: errorDetails,
    },
  });

  // Act & Assert
  let caughtError: any;
  try {
    await pollOperation.call(mockContext as IExecuteFunctions, 'operations/abc123', 10, 100);
  } catch (error) {
    caughtError = error;
  }

  expect(caughtError).toBeInstanceOf(NodeOperationError);
  expect(caughtError.message).toContain('Operation failed: Import source not found');

  // Verify error details are stringified
  if (caughtError.description) {
    expect(caughtError.description).toContain('fieldErrors');
    expect(caughtError.description).toContain('File not found in Files API');
  }
});
```

**Key Elements:**
- **Error object construction:** Create realistic error objects with messages and details
- **Exception verification:** Assert error type and message content
- **Nested error details:** Verify that detailed error information is preserved

## Best Practices

### 1. Test Isolation
- ✅ Each test should be independent
- ✅ Use `beforeEach` to reset mocks
- ✅ No shared state between tests
- ✅ Mock all external dependencies

**Example:**
```typescript
beforeEach(() => {
  jest.clearAllMocks();
  // Reset all mocks before each test
});
```

### 2. Clear Test Names
- ✅ Use descriptive names that explain what is being tested
- ✅ Include the condition and expected outcome
- ✅ Use "should" language

**Good:** `should import file with display name`
**Bad:** `test import`

### 3. Arrange-Act-Assert Pattern
- ✅ Always structure tests with three clear sections
- ✅ Arrange: Set up test data and mocks
- ✅ Act: Execute the function
- ✅ Assert: Verify the results

### 4. One Assertion Focus
- ✅ Each test should focus on one behavior
- ✅ Multiple assertions are OK if testing one concept
- ✅ Avoid testing multiple independent behaviors in one test

**Good:** Multiple assertions on same behavior
```typescript
expect(result).toEqual(expectedValue);
expect(mockFunction).toHaveBeenCalledWith(expectedParam);
expect(mockFunction).toHaveBeenCalledTimes(1);
```

### 5. Mock Carefully
- ✅ Mock external dependencies (API, file system, network)
- ✅ Don't mock the code under test
- ✅ Mock validators to test operation logic
- ✅ Use real implementations for utilities

### 6. Test Both Success and Error Paths
- ✅ Test happy path (success case)
- ✅ Test error cases
- ✅ Test edge cases
- ✅ Test validation failures

### 7. Use Type-Safe Mocks
- ✅ Cast mocks to correct types
- ✅ Verify mock call parameters
- ✅ Use `expect.objectContaining()` for partial matches
- ✅ Use `expect.any()` for flexible matching

**Example:**
```typescript
expect(apiClient.geminiApiRequest).toHaveBeenCalledWith(
  'POST',
  '/fileSearchStores',
  expect.objectContaining({
    displayName: 'Test Store',
  }),
);
```

### 8. Document Complex Test Logic
- ✅ Add comments explaining non-obvious setup
- ✅ Explain why certain mocks are needed
- ✅ Document edge cases being tested

```typescript
// Test stringList type to ensure metadata is properly parsed
// and comma-separated values are split into array
it('should include string list metadata in import', async () => {
  // ...
});
```

### 9. Use Test Fixtures Wisely
- ✅ Create reusable test data
- ✅ Keep fixtures up-to-date
- ✅ Document fixture structure

```typescript
// fixtures/documents.json
{
  "sampleDocument": {
    "name": "fileSearchStores/test/documents/doc1",
    "displayName": "Sample Document"
  }
}
```

### 10. Maintain Coverage
- ✅ Aim for >95% coverage
- ✅ Cover all branches (if/else)
- ✅ Cover error paths
- ✅ Review uncovered lines

## Lessons Learned

### Coverage Success Factors

1. **Comprehensive Scenario Testing**
   - Test basic operations, edge cases, and error paths
   - Test different parameter combinations
   - Test different data types (string, number, array, object)

2. **Mock-Based Testing**
   - Mocking external APIs enables isolated unit tests
   - Mocking validators allows testing operation logic independently
   - Sequential mock returns handle multi-parameter functions

3. **Pattern Consistency**
   - Following consistent patterns across tests improves maintainability
   - All operation tests follow Arrange-Act-Assert pattern
   - All validator tests follow similar structure

4. **Error Path Coverage**
   - Including error scenarios achieves high branch coverage
   - Testing both error presence and absence (with/without error.details)
   - Verifying error messages and descriptions

5. **Edge Case Testing**
   - String list parsing (comma-separated values)
   - Optional parameters (display name, metadata)
   - Empty results and null handling
   - Large file uploads
